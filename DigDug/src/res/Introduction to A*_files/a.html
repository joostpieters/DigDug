                  amitp Mod bobo 6 months ago

                  bobo, that's a really good question. It's something I didn't explain adequately. Both A* and Dijkstra's Algorithm can reduce the priority of a node as they explore and find better paths to the node. As long as a node is in the frontier, it can be updated with better priorities. In this case, A* hasn't found the best path to [3,1] (because it's efficientit's not spending as much time exploring paths to things that don't matter), and its cost_so_far is 16. It thinks the best path to [3,1] is by going west from [4,1]. If A* had time to explore more, it'd find a better path to [3,1] is by going north from [3,2], with a cost_so_far of 14. You can trigger this by blocking the shortest path and you'll see the costs drop.

                                                                                      amitp Mod Illuminaughty a month ago

                                                                                      Something more clever but maybe less practical: consider the number of steps remaining to be part of the state, along with location. There will be a movement edge from {x, y, steps} to {x1, y1, steps-1}, a stay still edge from {x, y, steps} to {x, y, steps-1}, and no edges from nodes {x, y, 0}. This allows you to use the edge weight for preferences, while using the nodes themselves to encode the constraint.

                                                                                                                                                amitp Mod Leo 4 months ago

                                                                                                                                                Yes, I considered implementing on the server side, but there are two big disadvantages:

                                                                                                                                                (1). It means I need a more complicated server, and have to deal with load spikes such as when the page is posted on reddit or hackernews and I get 50k hits all at once. Right now I have one "micro" amazon instance that serves static html, and it can handle all the traffic. With static html I also have the flexibility of using s3 or some other system. Static html can also be archived and cached; it's less fragile.

                                                                                                                                                (2). There's some latency for calculating results and getting them back. Unless I have servers all over the world, some users may have a latency of 300ms or more. That means as you drag the slider, I could not make a smooth animation of the recalculation. Right now with client side calculations it can run in under 10ms, which allows for smooth animations.

                                                                                                                                                It might seem like running the algorithms on the server would be useful if the algorithms were complicated, but because of problem (1), I would not be able to run the algorithms if there is a load spike! The total computing power in the browsers is larger than the total computing power on my servers. Running the algorithms on the server might be useful if the data is big, but so far these tutorials have small data. Maybe there will be a tutorial in the future where I need larger data.
